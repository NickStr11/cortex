# Subagents Playbook

> Правила делегирования задач субагентам. Читать перед запуском параллельных агентов.
> Обновлён: 2026-04-13. Источники: Anthropic docs (sub-agents), Certification, собственные уроки.

## Когда делегировать

| Ситуация | Действие |
|----------|----------|
| 3+ шагов ресёрча | → субагент |
| Независимые задачи | → параллельные агенты |
| Тяжёлый анализ (большие файлы, grep) | → субагент (экономия контекста) |
| Verbose output (тесты, логи, docs) | → субагент (изоляция вывода) |

### Когда НЕ делегировать (оставить в основном контексте)

- Частый back-and-forth с пользователем
- Несколько фаз с общим контекстом (план → код → тесты)
- Быстрое точечное изменение
- Критичная задержка (субагент тратит время на сбор контекста)

## Архитектура: Hub-and-Spoke

Главный агент = координатор. Субагенты общаются ТОЛЬКО с координатором, не друг с другом.

Координатор:
1. Чётко нарезает области ("ты ищешь X в директории A, ты — Y в директории B")
2. Видит все результаты и принимает решения
3. **Не доверяет выводам субагентов слепо** — верифицирует ключевые claims через прямой grep/read

## Кастомные субагенты (`.claude/agents/`)

Рекуррентные паттерны оформлены как файлы — не пишем промпт руками каждый раз.

| Агент | Модель | Назначение | Почему эта модель |
|-------|--------|-----------|-------------------|
| `researcher` | Sonnet | Ресёрч по коду, вебу, docs. Confidence-маркеры на каждый факт | Кейс: Hermes 22k vs реальные 126k stars поймал у Haiku. Sonnet кратно реже галлюцинирует на числах |
| `max-transcriber` | Sonnet | Транскрипция голосовых из Max | Пунктуация и расстановка фраз — Haiku мажет |
| `code-reviewer` | Sonnet | Ревью кода | Анализ диффа требует понимания контекста |
| `security-auditor` | Sonnet | Threat model verification | Безопасность — точность критична |
| `tg-digest-reader` | Haiku | Чтение TG-дайджестов с VM | Просто читает текст из journalctl, не интерпретирует. Haiku достаточно |
| `architect` | Opus | Архитектура и стек | Сложные трейдоффы, нужна максимальная глубина |

> **2026-05-01 пересмотр:** изначально 2026-04-30 поднял всех Haiku→Sonnet «потому что квота недогружена». Ответный отзыв (Кент-агент) указал что это финансовый, а не качественный аргумент. Пересмотрел per-agent: `researcher`/`code-reviewer`/`max-transcriber`/`security-auditor` остаются на Sonnet (есть конкретные кейсы где Haiku давал плохие результаты). `tg-digest-reader` откатился на Haiku — он реально только читает текст. Если квоты Sonnet кончатся — researcher держим, остальных на Haiku.

### Разделение Claude vs Codex

- **Claude = orchestrator + interactive** — диалог, правки, рутина (research, transcribe)
- **Codex = heavy specialist** — adversarial audits, repo mapping, browser debugging
- Дубли убраны (2026-04-29): `security-auditor` и `tg-digest-reader` из `.claude/agents/` — закрыто Codex.security_reviewer и инлайновой командой

### Как использовать

- Автоматически: Claude сам выбирает по `description`
- Явно: `@agent-name` в промпте
- Сессия: `claude --agent code-reviewer`

### Persistent Memory

Агенты с `memory: project` копят знания в `.claude/agent-memory/`. Между сессиями помнят паттерны, решения, воркэраунды. Сейчас включено для: `code-reviewer`, `max-transcriber`.

## Model Routing

| Задача | Модель | Почему |
|--------|--------|--------|
| Поиск по коду, grep, чтение | Sonnet | Точнее цитирует, не путает строки |
| Ревью кода, анализ | Sonnet | Баланс скорость/качество |
| Ресёрч с числами/сравнениями | Sonnet | Кратно меньше галлюцинаций на цифрах |
| Генерация кода, сложная логика | Sonnet/Opus | Нужно качество |
| Архитектурные решения | Opus | Максимальная глубина |

Sonnet — дефолт для всех субагентов. Opus только для архитектуры. Haiku не используется (квота юзера на Sonnet недогружена, экономия не актуальна, точность важнее).

## Промпт для субагента

Хорошо:
```
Найди все вызовы process_refund в tools/. Верни JSON:
{"files": [...], "count": N, "has_error_handling": bool}
```

Плохо:
```
Посмотри как у нас устроены рефанды
```

Правила:
- Конкретная задача, конкретный формат ответа
- Структурированный вывод (JSON/список) > стена текста
- Указывать scope (директория, файлы) чтобы агент не бродил по всему репо
- Объяснить контекст — субагент не видит основной разговор

## Обработка ошибок

1. **Локальное восстановление** — субагент сначала пробует сам (retry, альтернативный путь)
2. **Эскалация с контекстом** — если не получилось, возвращает: что пробовал, что не сработало, почему
3. **Graceful degradation** — частичный результат лучше чем "не смог". Пометить пробелы явно

## Антипаттерны (из опыта)

| Проблема | Урок |
|----------|------|
| Субагент сказал "API не реализованы" — были в server.py | Верифицировать claims через прямой grep/read (2026-03-11) |
| Субагенты дублируют поиск по одним файлам | Координатор должен разделять scope явно |
| Субагент вернул 500 строк анализа | Требовать структурированный вывод, ограничивать объём |
| Каждый раз писали промпт для Max транскрипции | Оформить как кастомный агент с memory (2026-04-13) |
| Субагент не знал про Chrome CDP flow | Вшить domain knowledge в system prompt агента (2026-04-13) |
| Researcher упал `Prompt is too long` на списке 72 элементов | Большие списки разбивать на батчи в самом промпте; либо несколько последовательных Task с фрагментами; либо переключиться на Exa-поиск напрямую (2026-04-29) |
| Я делал TG digest руками через `journalctl`, не через `tg-digest-reader` | Триггер "посмотреть лог на VM" / "прочитать дайджест" → СРАЗУ субагент (2026-04-29) |

## Lesson capture — auto

`SubagentStop` hook (`.claude/hooks/subagent-stop.py`) автоматически пишет summary каждого завершённого субагента в `.claude/agent-memory/<agent>/lessons.md` (append-only). Содержит: дату, описание задачи, tool uses, prompt-preview, ошибки, итоговый текст. Используй для чтения истории работы конкретного агента — например что `researcher` уже искал на этой неделе и какие приёмы работали.

Перед запуском агента — проверь `.claude/agent-memory/<agent>/lessons.md` на повторяющиеся паттерны.

## Параллелизм

- Независимые задачи → запускать одновременно (несколько Agent tool calls в одном сообщении)
- Зависимые → последовательно, результат первого → промпт второго
- `run_in_background: true` для задач где не нужен результат немедленно
- Для 3+ связанных задач с координацией → TeamCreate (мощнее параллельных Agent)

## MCP Server Scoping

Тяжёлые MCP серверы (Playwright, browser-use) — скоупить на конкретных агентов через `mcpServers` в frontmatter. Не грузить tool descriptions в основной контекст.

**disallowedTools > tools для агентов с MCP.** Белый список (`tools: [Read, Grep]`) не видит MCP-инструменты с длинными именами (`mcp__exa__web_search_exa`). Если нужен доступ ко всем MCP — используй чёрный список (`disallowedTools: [Edit, Write]`), чтобы не перечислять каждый MCP-тул вручную.

## Codex CLI MCP как субагент

- `reasoningEffort: xhigh` для серьёзных задач
- Хорош для: второе мнение, ресёрч, websearch
- Модели: gpt-5.3-codex (дефолт), o3 для сложной логики
