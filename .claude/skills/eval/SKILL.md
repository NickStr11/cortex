# Eval — Post-Session Self-Evaluation

Запускается в конце сессии (перед /handoff) или по запросу `/eval`.
Оценивает качество работы, фиксирует инциденты, предлагает design injections.

## Когда запускать

- Перед `/handoff`
- После завершения крупной задачи
- Когда пользователь говорит "оцени" / "что пошло не так" / "eval"

## Процесс

### Шаг 1: Сбор контекста

Прочитай:
- Текущий чат — что делали, какие ошибки были
- `docs/incidents.md` — последние инциденты
- `memory/MEMORY.md` — паттерны и уроки

### Шаг 2: Оценка по 4 осям

Оцени сессию по шкале 1-5:

| Ось | Что оцениваем | 5 = идеально | 1 = провал |
|-----|---------------|--------------|------------|
| **Reasoning** | Логика решений, выбор подхода | Сразу нашёл корень, минимум итераций | 5+ попыток, брутфорс, не понял проблему |
| **Rules** | Следование CLAUDE.md, стандартам | Все правила соблюдены | Нарушил инварианты, сломал production |
| **Code** | Качество кода, тесты, типы | Чистый, типизированный, протестирован | Хаки, без тестов, сломал билд |
| **Efficiency** | Экономия контекста, делегирование | Субагенты, параллельность, минимум шагов | Всё руками, раздул контекст, повторы |

### Шаг 3: Фиксация инцидентов

Если в сессии были ошибки (build fail, неверный подход, сломанный код):

1. Для каждой ошибки — записать инцидент в `docs/incidents.md` по формату
2. Провести 5 Почему (root cause analysis)
3. Предложить **design injection** — конкретное изменение в стандартах/скиллах/хуках

### Шаг 4: Design Injection

Если найдена повторяющаяся проблема:
- Обновить `CLAUDE.md` если это правило поведения
- Обновить `docs/integrations/*.md` если это знание об интеграции
- Добавить в `memory/MEMORY.md` если это паттерн между сессиями
- Создать/обновить хук если это можно автоматизировать

### Шаг 5: Отчёт

Формат вывода:

```
## Eval — [дата]

### Scores
| Reasoning | Rules | Code | Efficiency |
|-----------|-------|------|------------|
| X/5       | X/5   | X/5  | X/5        |

### What went well
- ...

### Incidents (if any)
- INC-...: описание → записано в docs/incidents.md

### Design Injections (if any)
- Обновлён [файл]: что поменялось

### Lessons
- ...
```

## Антипаттерны

- НЕ завышать оценки — честная оценка ценнее приятной
- НЕ пропускать инциденты — даже мелкие ошибки фиксировать
- НЕ делать design injection на каждый чих — только повторяющиеся проблемы
- НЕ писать "всё было хорошо" если были косяки
